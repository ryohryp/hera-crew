### 第1章：14B「統一」の罠（約1,200字）

**事故の症状**
- DeepSeek-R1:14b にマネージャー役を任せると、出力 JSON が `<think>...</think>` で汚染されてパース不能になる
- qwen2.5:14b 統一だと、コーディング精度が qwen2.5-coder:14b に劣る
- 14B × 3 同時ロードはVRAM 16GBでは不可能（実測 OOM）

**設計変更**
| Step | 旧構成（前記事） | 新構成（現在） | 理由 |
|------|-----------------|---------------|------|
| Step1 Thinker | qwen2.5:14b | **qwen2.5-coder:14b (9GB)** | タスク分解＝コード読解力が要る |
| Step2 Critic | deepseek-r1:14b | **qwen3:8b (5.2GB)** | 判定だけなら 8B で十分。swap時間短縮 |
| Step3 Manager | qwen2.5-coder:14b | **deepseek-r1:14b (9GB)** | 実装は深い思考が要る |
| Step4 Verifier | deepseek-r1:14b | **deepseek-r1:14b 再利用** | swap せず再ロード回避 |

**学び**
- 「14Bで揃える」は美しいが運用の答えではない。**各Stepの仕事に応じてモデルを変えるのが正解**
- Ollama の swap（モデル切替）は思ったより速い。各時点1モデルで回すと VRAM が圧倒的に余る
- `<think>` を出すモデルは「思考用」、出さないモデルは「応答用」と用途分離

**コード断片**: `config/llms.yaml` のbefore/after

### 第2章：早期終了ガード設計（約1,300字）

**事故の症状**
- 「Hello」の一言を投げたら、Thinker が真面目にタスク分解を始め、5分かかって「挨拶を返す」と結論
- 「ありがとう」に対して、Step4 が「より丁寧な感謝の表現を検討すべき」と改善指示

**根本原因**
- LLMに「これは簡単なタスクですか？」と聞くと、ほぼ全部「分解が必要」と答える（自分の仕事を否定しないバイアス）
- LLM自己判定はSIMPLE shortcutに使えない

**解決策：機械的ガード**
```python
def _is_simple_eligible(self, task: str) -> bool:
    if len(task) > 50:                       return False  # 長文はNG
    if any(kw in task for kw in FORBIDDEN):  return False  # 「実装」「作成」「書いて」「ファイル」「コード」等
    if re.match(r'\d+\.\s', task):           return False  # 番号付きリストはNG
    if '\n' in task:                          return False  # 改行はNG
    return True
```

**結果**
- Hello: **649秒 → 16.4秒（39倍速）**
- 「ありがとう」「OK」「もう一度」等の短文応答が即返る

**学び**
- LLM自己判定 < 決定的ロジック。**判断を「LLMの仕事」と「コードの仕事」で分離する**
- 早期終了は「速さ」より「ノイズの除去」のため。簡単な質問にまで4-stage回すと履歴がゴミだらけになる

**画像案**: 早期終了パスのフローチャート（Mermaid）

### 第3章：Step4 検証エージェントの罠（約1,400字）

**事故の症状①：誤訳事故**
- Step3 が `convert_to_snake_case()` という関数を生成
- Step4 が「**蛇足記法（snake_case）への変換は冗長です**」とコメント → リジェクト
- 実際は単に "snake_case" を「蛇足記法」と意訳しただけ。検証エージェントが日本語に引きずられた

**事故の症状②：採点バイアス**
- Step4 の評価が「妥当」「READY_TO_DELIVER」に偏る
- Step3 が明らかにバグっているコードを書いても LGTM
- 自己採点（同じ deepseek-r1 が Step3 と Step4 を両方やる）の弊害

**現状の対処**
- 採点ラベルを `NEEDS_REVISION / READY_TO_DELIVER / LGTM` の3段に分け、最甘判定を機械的にカウント
- HTML レポートで Step4 のコメントを **赤字ハイライト**して人間が必ずチェック

**未解決の課題（次に潰す）**
- Step4 を別モデル（例：gemma3-27b の量子化版）に分離
- 「英単語をそのまま英語で読む」プロンプト指示の追加
- カスタム評価ルーブリック（採点表）の事前提示

**学び**
- 検証エージェントは **「実装と異なるモデル」「異なる視点」「決定的なルーブリック」の3点セットでないと機能しない**
- 日本語LLMの誤訳は侮れない。`snake_case`→蛇足記法、`backend`→裏口、などの実例

### 第4章：トークン浪費の罠（約1,000字）

**事故の症状**
- 1タスクで合計 22,000トークン消費（FastAPI JWT実装）
- うち約 40% が deepseek-r1 の `<think>...</think>` 連鎖
- ローカル実行なので金銭コストはゼロだが、**VRAM占有時間が伸びる＝待ち時間が伸びる**

**内訳の図示（円グラフ案）**
- Step1 Thinker: 3,200 tok (15%)
- Step2 Critic: 1,800 tok (8%)
- Step3 Manager: 11,000 tok (50%) ← 大半が <think>
- Step4 Verifier: 6,000 tok (27%) ← これも <think>多め

**対策候補（未着手）**
- システムプロンプトに `Do not use <think> tags. Output final answer directly.`
- `num_predict` を低めに固定
- LiteLLM の post-process で <think>...</think> を strip

**学び**
- 「無料だから消費していい」ではない。**思考時間 × VRAM占有 = 開発体験の質**
- ローカルLLMでも「節約」の概念は要る。ただし金額ではなく秒数で

### 第5章：スコープ依存の罠（約900字）

**事故の症状**
- Step3 が Ollama タイムアウトで失敗 → `manager_session` 変数が未定義
- Step4 で `NameError: name 'manager_session' is not defined` で全パイプライン停止
- せっかく Step1〜2 の出力があるのに、**全部捨てて最初からやり直し**

**設計変更**
- 各Stepの出力を **`reports/step_outputs_<timestamp>.json`** に逐次保存
- Step4 が壊れても Step1〜3 の出力は救出可能
- 次回再開時に「Step3から再開」が選べるよう、状態をディスクに永続化

**コード断片**
```python
# 各Step終了時に必ず呼ぶ
def _persist_step(self, step_id: int, output: str):
    path = f"reports/step_outputs_{self.run_id}.json"
    data = json.load(open(path)) if os.path.exists(path) else {}
    data[f"step_{step_id}"] = output
    with open(path, 'w', encoding='utf-8') as f:
        json.dump(data, f, ensure_ascii=False, indent=2)
```

**学び**
- LLMパイプラインは**ノードが落ちる前提で組む**。エンタープライズの ETL と同じ
- 「途中で死んでも、そこまでの成果は失わない」が最低ライン

### 第6章：可観測性の三層構造（約1,200字）

**設計**
| ファイル | 振る舞い | 用途 |
|---------|---------|------|
| `reports/hera_report.html` | 毎回上書き | 直近1回の可視化（人間が見る） |
| `reports/history.jsonl` | 追記のみ | 全実行の集計（傾向分析） |
| `reports/step_outputs_<ts>.json` | 実行ごと別ファイル | Stepごとの本文（救出用） |

**なぜ三層に分けたか**
1. **HTML 上書き**：見たい時はいつも「最新の1回」。古いHTMLは情報過多でノイズ
2. **JSONL 追記**：1行=1実行のサマリ。`jq` や Pandas で統計が取れる
3. **JSON 個別保存**：Step ごとの **完全な本文**。Step4 が `<think>` 暴走で巨大化した時も、Step1〜3 は無事

**HTML レポートのスクショ案**
- Token 内訳のドーナツチャート
- Step ごとの所要時間バーチャート
- 各Stepの出力（折りたたみ可）

**学び**
- 可観測性は「壊れた時に何が見えるか」が全て
- LLM運用は **デバッグの8割が「何が起きたか分からない」との戦い**。ログが命

### 第7章：開発環境の罠（約1,000字）

**罠①：Cowork Edit/Write の末尾切断バグ**
- 約 10KB 超のファイルを Edit/Write すると、UTF-8 マルチバイト境界を無視して**末尾が切れる**
- yaml ファイルや Python ソースが途中で `def foo(` のまま終わって SyntaxError

**回避策**
```bash
# bash heredoc 経由で Python の atomic write を実行
python3 <<'EOF'
content = """..."""
import os, tempfile
fd, tmp = tempfile.mkstemp(dir='I:\\04_develop\\hera-crew')
with os.fdopen(fd, 'w', encoding='utf-8') as f:
    f.write(content)
os.replace(tmp, 'target.py')
EOF
```

**罠②：PowerShell の `Get-Clipboard | python` パイプ**
- 複数行プロンプトをクリップボード経由で渡すと、改行が `\r\n`/`\n` 混在で壊れる
- 特に日本語が含まれると Shift_JIS↔UTF-8 で文字化け

**回避策**
- 長文プロンプトは **必ずファイル化**して `tools/run_hera.py --file <path>` で渡す

**学び**
- ツール固有のバグは「ハマってから知る」では遅い。**プロジェクト初期に運用ルールとして CLAUDE.md に書いておく**
- 環境の罠は技術ではなく**運用設計**で殺す
