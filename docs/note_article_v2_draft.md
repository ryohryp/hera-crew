# 【続編・失敗学】ローカル14Bマルチエージェントを1ヶ月運用してわかった、9つの罠と対処

## はじめに：あの記事から1ヶ月、現実は甘くなかった

2026年4月1日、私は[こちらの記事](https://note.com/yakinbo/n/n8674d2948506)で「VRAM 16GBで最強のローカル14Bマルチエージェントを構築した話」を投稿しました。CrewAIベースの3役（マネージャー / レビュアー / テックリード）構成、すべて14Bモデルで統一、クラウドへのフォールバックも視野に入れた華々しい設計でした。

あれから約1ヶ月。私のhera-crewは、当時とは似ても似つかない姿に進化しました。

なぜか。それは、**「動いた」と「使える」の間に、想像していたより深い谷があった**からです。

タスクを投げると確かに動く。しかし「Hello」の一言に5分かけて分解を始めたり、検証エージェントが英単語を珍訳してリジェクトしたり、Step3が落ちた瞬間にStep1〜2の出力もろとも全消滅したり——巷のローカルLLM紹介記事は「動いた／速かった」で終わりますが、その先に広がる運用の地獄については、誰もあまり書いてくれません。

この記事は、その**地獄の博物館**です。

ローカルLLMでマルチエージェントを動かしてみたい方、あるいはすでに動かして「なんかうまくいかない」と感じている中級者の方に向けて、私が1ヶ月で踏んだ9つの地雷と、そこから出した暫定回答を共有します。

なお、本記事で扱うのはhera-crewの**現行版（2026年4月末時点）**です。前記事のCrewAI構成は事実上廃棄しました。理由は本文で順に明かしていきます。

## 第1章：14Bモデル「統一」の罠 — なぜ役割別分離に舵を切ったか

前記事で私は「14Bで揃えるのが美しい」と書きました。あれは半分正解で、半分間違いでした。

### 事故の症状

**症状①：DeepSeek-R1のJSON汚染**

マネージャー役にdeepseek-r1:14bを据えると、関数呼び出しのJSON出力が `<think>...</think>` ブロックに埋もれてパースが通りません。前記事でも軽く触れましたが、当時はqwenをマネージャーに据えて回避するつもりでした。しかし——

**症状②：qwen2.5:14b統一だとコーディング精度が落ちる**

実装をqwen2.5:14bに任せると、qwen2.5-coder:14bと比べて型ヒントの抜け、import漏れ、エッジケースの扱いが明らかに荒い。コーディング特化モデルとの差は無視できないレベルでした。

**症状③：14B × 3 同時ロードはVRAM 16GBに収まらない**

これは机上の計算でも怪しかったのですが、実機で試したら案の定OOM。Ollamaは複数モデルを並走させるとそれぞれを9GB前後抱えるため、VRAM 16GBでは2モデルすら共存させられません。

### 設計変更：役割別 × Ollama swap前提

そこで現行版では、**各時点1モデルしかVRAMに乗せない**前提で、Stepごとに最適なモデルへ切り替える方式に組み替えました。

| Step | 旧構成（前記事） | 新構成（現在） | 役割 |
|------|----------------|---------------|------|
| Step1 Thinker | qwen2.5:14b | **qwen2.5-coder:14b (9GB)** | タスク分解 / 早期終了判定 |
| Step2 Critic | deepseek-r1:14b | **qwen3:8b (5.2GB)** | LOCAL/FALLBACK判定 |
| Step3 Manager | qwen2.5-coder:14b | **deepseek-r1:14b (9GB)** | 実行・コード生成 |
| Step4 Verifier | deepseek-r1:14b | **deepseek-r1:14b 再利用** | 批判的検証 |

ポイントは3点です。

ひとつ目は、Step1に**コーディング特化モデル**を当てたこと。タスク分解はコードの構造を読む仕事なので、qwen2.5-coderが圧倒的に向いています。

ふたつ目は、Step2のCriticを**8Bに落とした**こと。判定だけならMarkdown表を出させる程度の知性で十分で、サイズを下げるとswap時間が短縮されます。

みっつ目は、Step3とStep4で**同じdeepseek-r1を再利用**すること。Verifierでもう一度モデルをロードし直すのは無駄なので、Step3で温まったdeepseek-r1をそのままStep4で叩く設計にしました。

### 学び

- 「14Bで揃える」は美しいが、運用の答えではない
- Ollamaのswap（モデル切替）は思ったより速い。**各時点1モデルで回すとVRAMが余り、設計の自由度が増える**
- `<think>` を出すモデルは「思考用」、出さないモデルは「応答用」。**用途で分けてマネージャーには応答用を据える**

設定ファイル（`config/llms.yaml`）で各Stepへのモデル割り当てを単一ソース化したので、モデル変更は1ファイル書き換えるだけで済みます。これも運用上の救いでした。

## 第2章：早期終了ガード設計 — LLMに「簡単？」と聞いてはいけない

これは私が一番苦笑いした罠です。

### 事故の症状

「Hello」とだけ投げました。すると、Thinkerが大真面目にタスク分解を始めました。

```
Step1: 「ユーザーの意図を分析する」
Step2: 「適切な挨拶を選択する」
Step3: 「文化的背景を考慮する」
Step4: 「応答文を生成する」
```

5分かかって、最終的に「Hello」と返ってきました。

別の日には「ありがとう」と入れたら、Step4が「**より丁寧な感謝の表現を検討すべきです**」とコメントをつけてリジェクトしてきました。

### 根本原因

LLMに「これは簡単なタスクですか？」と聞くと、**ほぼ全部「分解が必要」と答えます**。理由は単純で、モデルにとって「自分の仕事を否定する」のはバイアス的に難しいからです。営業マンに「これは営業不要ですか？」と聞くようなもので、構造的に答えは決まっています。

つまり、**SIMPLE shortcut判定にLLM自己判定は使えません**。

### 解決策：機械的ガード

決定的なロジックで判定するしかない、と腹をくくって書いたのが以下のガードです。

```python
def _is_simple_eligible(self, task: str) -> bool:
    if len(task) > 50:
        return False
    if any(kw in task for kw in FORBIDDEN_KEYWORDS):
        # FORBIDDEN: 実装 / 作成 / 書いて / ファイル / コード / 修正 / リファクタ / ...
        return False
    if re.match(r'\d+\.\s', task):  # 番号付きリストはNG
        return False
    if '\n' in task:  # 改行を含む長文はNG
        return False
    return True
```

50字を超えたらNG、コード生成系のキーワードを含んだらNG、番号付きリストや改行を含んだらNG。**LLMに考えさせず、コードで切る**。

### 結果

| プロンプト | 旧（gemma4:26b統一） | 現行（早期終了あり） | 倍率 |
|----------|---------------------|--------------------|------|
| Hello | 649秒 | **16.4秒** | 39倍速 |
| ありがとう | 約8分 | 約12秒 | 40倍速 |
| OK | 約5分 | 約10秒 | 30倍速 |

「Hello」が16秒で返ってくるようになっただけで、対話のリズムが激変しました。

### 学び

- **判断を「LLMの仕事」と「コードの仕事」で分離する**。意思決定の一部はコードに任せた方が速くて確実
- 早期終了は「速さ」のためだけではない。**簡単な質問にまで4-stage回すと履歴がゴミだらけになる**ノイズ問題を解決するため

副次効果として、history.jsonlの実行ログが「意味のある実行」だけで埋まるようになり、傾向分析の精度も上がりました。

## 第3章：Step4 検証エージェントの罠 — 「snake_case」を「蛇足記法」と訳す日

ここが最も笑えて、最も深刻な罠です。

### 事故の症状①：日本語誤訳によるリジェクト

ある日、Step3が `convert_to_snake_case()` という関数を生成しました。これに対するStep4のコメントが、こうです。

> 「**蛇足記法（snake_case）への変換は冗長であり、不要なステップを増やしています。再考が必要です。**」

私は二度見しました。

`snake_case` を「蛇足記法」と意訳した上で、「冗長」というニュアンスを汲んでリジェクトしてきたのです。Step4は日本語ベースのプロンプトで動いているので、英単語を**律儀に意訳して文意を引きずる**ことがあるのです。

他にも `backend` → 「裏口」、`fallback` → 「後退」、`override` → 「上書き支配」など、誤訳の博物館が築かれつつあります。

### 事故の症状②：採点バイアス

もうひとつの根深い問題が、Step4の採点が**「妥当」「READY_TO_DELIVER」に偏る**ことです。

Step3が明らかにバグっているコード（例：戻り値が型ヒントと不整合、必須引数の欠落）を出しても、Step4がLGTMを出すケースが頻発しました。

原因は明白です。Step3とStep4は**同じdeepseek-r1:14b**を再利用しています。つまり**自己採点**になっており、自分の生徒を自分で評価する構図です。甘くなって当然でした。

### 現状の対処（暫定）

採点ラベルを `NEEDS_REVISION / READY_TO_DELIVER / LGTM` の3段に分けた上で、HTMLレポートでStep4のコメントを**赤字ハイライト**にして、人間が必ず目視チェックするようにしました。完全な自動化は諦め、Human-in-the-Loop に倒した形です。

### 未解決の課題（次に潰す）

- Step4を別モデル（gemma3-27bの量子化版や、qwen3-32bなど）に分離して**他者採点**に切り替える
- システムプロンプトに「英単語はそのまま英語で読み、意訳しない」と明記する
- 採点ルーブリック（型チェック / import完全性 / エッジケース網羅）を**事前に提示**して、評価軸をブレなくする

### 学び

検証エージェントが機能するには、**「実装と異なるモデル」「異なる視点」「決定的なルーブリック」の3点セット**が要ります。どれか欠けると、検証は儀式になります。

特に**自己採点は信用してはいけない**。これはLLMだけでなく、組織でもよく言われる話ですね。

## 第4章：トークン浪費の罠 — deepseek-r1の `<think>` 連鎖

ローカル実行なので金銭コストはゼロです。なのにトークンの話を書くのには理由があります。

### 事故の症状

FastAPI + JWT認証の実装タスクを投げたところ、合計**22,000トークン**を消費しました。内訳は概ね以下の通りです。

| Step | モデル | 消費トークン | 占有率 |
|------|--------|-------------|--------|
| Step1 Thinker | qwen2.5-coder:14b | 約3,200 tok | 15% |
| Step2 Critic | qwen3:8b | 約1,800 tok | 8% |
| Step3 Manager | deepseek-r1:14b | **約11,000 tok** | 50% |
| Step4 Verifier | deepseek-r1:14b | 約6,000 tok | 27% |

Step3とStep4で合わせて77%。そしてその大半が `<think>...</think>` の**思考連鎖**で埋まっています。実際の出力（コードや評価コメント）は2割程度で、残り8割が「考えているところ」のテキストです。

### なぜこれが問題か

ローカル実行なのでAPI料金は発生しません。しかし——

- **トークン量 ∝ VRAM占有時間 ∝ 待ち時間**

つまり、`<think>` が長引くほど、ユーザーの体感が悪くなります。22,000トークンの生成には7〜10分かかっており、そのうち5分以上が思考の独白で消えています。

### 対策候補（未着手）

- システムプロンプトに `Do not use <think> tags. Output the final answer directly.` を追加
- `num_predict` を保守的な値（例：2048）で固定
- LiteLLMのpost-processフックで `<think>...</think>` をstripしてからプロンプトに渡す

### 学び

「無料だから消費していい」ではないのです。**思考時間 × VRAM占有 = 開発体験の質**。ローカルLLMでも「節約」の概念は要ります。ただし金額ではなく**秒数**で。

これは次に必ず潰します。

## 第5章：スコープ依存の罠 — Step3が転んだらStep4も道連れ

これは私のコーディングミスでもありますが、設計上の教訓として書き残します。

### 事故の症状

ある日、Step3の実行中にOllamaがタイムアウト（120秒で切れる設定でした）。Step3は途中で死んだのですが、Step4に遷移しようとしたところ——

```
NameError: name 'manager_session' is not defined
```

Step3で初期化されるはずだった `manager_session` 変数が未定義のまま、Step4のコードが参照しに行ってクラッシュ。**Step1〜2の出力を含めて、すべてが闇に消えました**。

### 設計変更：状態のディスク永続化

LLMパイプラインは**ノードが落ちる前提で組む**べきだ、という基本に立ち返りました。エンタープライズのETLでは当たり前ですよね。

各Stepの出力を、終了時に必ずディスクへ吐き出すようにしました。

```python
def _persist_step(self, step_id: int, output: str):
    path = f"reports/step_outputs_{self.run_id}.json"
    data = json.load(open(path)) if os.path.exists(path) else {}
    data[f"step_{step_id}"] = output
    with open(path, 'w', encoding='utf-8') as f:
        json.dump(data, f, ensure_ascii=False, indent=2)
```

これで、Step4が壊れてもStep1〜3の出力は救出可能になりました。次回はStep3から再開する手段（再開モード）も実装予定です。

### 学び

- **「途中で死んでも、そこまでの成果は失わない」**を最低ラインに置く
- スコープ依存は楽だが、**LLMタスクは10回に1回は落ちる前提**で書くべき
- 状態管理は明示的に。共有変数で繋ぐのではなく、ディスクのJSONを真とする

## 第6章：可観測性の三層構造 — 「壊れたときに何が見えるか」を最優先で設計する

LLM運用で最も時間を吸われるのは、**「何が起きたか分からない」との戦い**です。デバッグの8割がこれです。

そこでhera-crewでは、出力を3層に分けて記録しています。

| ファイル | 振る舞い | 用途 |
|---------|---------|------|
| `reports/hera_report.html` | 毎回上書き | 直近1回の可視化（人間が見る） |
| `reports/history.jsonl` | 追記のみ | 全実行の集計（傾向分析） |
| `reports/step_outputs_<ts>.json` | 実行ごと別ファイル | Stepごとの完全な本文（救出用） |

### なぜ三層に分けたか

**HTML（上書き）**：見たいのはいつも「最新の1回」です。古いHTMLが残っていると、ファイル一覧がノイズで埋まります。Token内訳のドーナツチャート、Stepごとの所要時間バーチャート、各Stepの出力（折りたたみ可）を一画面に収めて、人間が3秒で状況を把握できる作りにしました。

**JSONL（追記のみ）**：1行=1実行のサマリ。`jq` やPandasで集計しやすく、傾向分析（どのStepで時間を食っているか、トークンの推移はどうか）に使えます。

**Per-step JSON（個別保存）**：Stepごとの**完全な本文**を別ファイルにします。第5章で書いた通り、Step4が `<think>` 暴走で巨大化したり、HTMLレンダリングが何かの理由で壊れても、Step1〜3の本文は無事です。

### 学び

可観測性は**「壊れた時に何が見えるか」**で評価すべきです。「動いている時にきれいなダッシュボードが見える」のは二の次。失敗時に手元に残る情報量こそが、運用の生死を分けます。

LLM運用は、**普通のWebシステムよりも壊れやすい**前提で組むべきです。これは1ヶ月でしみじみ学びました。

## 第7章：開発環境の罠 — Cowork Edit末尾切断とPowerShellパイプ

ここからは**ツール側の罠**です。LLM本体ではなく、開発環境固有の落とし穴の話です。

### 罠①：Cowork Edit/Writeの末尾切断バグ

私はAnthropicのCoworkモードで開発作業の一部を行っているのですが、ある日**ファイル末尾が突然欠ける現象**に出くわしました。

具体的には、Pythonファイルが `def foo(` で唐突に終わっており、SyntaxError。yaml ファイルも途中で `model:` だけ残っていて壊滅。

調べると、約10KBを超えるファイルをEdit/Writeで書き出すと、**UTF-8マルチバイト境界を無視して末尾が切れる**バグがあるようでした。日本語コメントが多いファイルほど影響が大きい。

#### 回避策：bash heredoc経由のアトミック書き込み

bash経由でPythonスクリプトを実行し、tempfileに書いてから `os.replace` でアトミックに置き換える。これで末尾切断は完全に防げます。具体的には、bash heredoc内で `tempfile.mkstemp` → `f.write(content)` → `os.replace(tmp, target)` の順で書き出すパターンです。

### 罠②：PowerShellの `Get-Clipboard | python` パイプ

長文プロンプトをクリップボード経由でhera-crewに渡す運用を試したのですが、複数行の場合**改行が `\r\n` / `\n` 混在で壊れる**現象に遭遇しました。日本語が含まれていると、Shift_JIS と UTF-8 のせめぎ合いで文字化けすることもあります。

#### 回避策：必ずファイル化

```powershell
# NG: クリップボード経由
Get-Clipboard | python tools/run_hera.py

# OK: ファイル経由
python tools/run_hera.py --file prompts/long_task.txt
```

地味ですが、安定します。

### 学び

- ツール固有のバグは「ハマってから知る」では遅い。**プロジェクト初期にCLAUDE.mdなどの運用ドキュメントへ書き留めておく**
- 環境の罠は技術ではなく、**運用ルールで殺す**
- 「便利そう」なパイプ経路は、長文と日本語の前で意外と簡単に壊れる

## 第9章：次に潰す課題

完成にはほど遠く、まだ罠は山積みです。直近で潰す予定の課題を共有します。

- **Step4採点の独立化**：別モデルへの分離 / カスタムルーブリックの事前提示。自己採点バイアスの根治
- **`<think>` 抑制実験**：システムプロンプトとnum_predictで思考連鎖を圧縮、トークン浪費を半減させたい
- **`run()` メソッドのリファクタ**：現在約160行と肥大化しており、責務分離が必要
- **Step3→Step4の状態管理**：NameErrorリスクを構造的に潰す。状態マシン化を検討
- **Step4日本語誤訳対策**：英単語の引用ルール明文化、専門用語の固有名詞扱い

これらは次の3ヶ月で順次手を入れていきます。
